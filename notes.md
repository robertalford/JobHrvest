## Notes on potential approach(es) to achieve goal.

1. Crawling Infrastructure (Getting the HTML)
   **Conventional approach:** Scrapy (Python) is the gold standard for open-source crawling. It handles request queuing, rate limiting, retries, politeness (robots.txt), and middleware out of the box. For JavaScript-rendered pages (which many modern job boards are), pair it with Playwright or Selenium via scrapy-playwright. Playwright is strongly preferred over Selenium nowadays — it's faster, more reliable, and has better async support.
   **More radical:** Consider using a headless browser pool managed by something like Browserless (open-source, self-hostable). This gives you full browser rendering at scale without managing individual browser instances. Another option is crawl4ai — a relatively new open-source crawler specifically designed for AI-friendly extraction. It handles JS rendering and outputs clean markdown, which feeds nicely into LLM-based extraction later.
   **Key design decision:** You'll hit two categories of sites — static HTML (easy, Scrapy alone works) and SPA/JS-rendered (needs a real browser). Build your crawler to try the lightweight path first, then fall back to full browser rendering. This saves enormous resources at scale.
   For storage of raw HTML, use a combination of S3-compatible object storage (MinIO is the self-hosted open-source option) for raw page snapshots and PostgreSQL for structured metadata. Keep every raw snapshot — you'll want to re-process pages later as your extraction logic improves.
            
2. Careers Page Discovery (The First Hard Problem)
   This is where it gets interesting. Given a company's root URL, how do you find where their jobs live?
   **Heuristic approach (start here):** Build a multi-signal scorer that combines URL pattern matching (paths containing /careers, /jobs, /opportunities, /vacancies, /work-with-us, /join, /openings and their variations), link text analysis (anchor text containing "careers", "jobs", "we're hiring", etc.), and navigation structure analysis (careers links tend to appear in footer navs or top-level menus). Crawl the site to a shallow depth (2-3 levels) and score every discovered page. This gets you 70-80% accuracy on its own.
   **LLM-augmented approach (the real lever):** After your heuristic pass narrows candidates, send the page content (or a cleaned/condensed version) to a local LLM and ask it to classify whether this is a careers/jobs page. You can use Ollama to run models like Llama 3 or Mistral locally for free. The prompt is straightforward: given the page title, meta description, heading text, and a sample of body text, classify as careers-page / not-careers-page / ambiguous. This pushes accuracy into the 90%+ range.
   **Innovative approach:** Train a lightweight classifier (a fine-tuned BERT or even a simple TF-IDF + logistic regression model) on a labeled dataset of careers vs. non-careers pages. You can bootstrap this dataset from your initial LLM classifications, then human-verify a sample. Once trained, this runs orders of magnitude faster than LLM inference and can handle your scaling needs. Hugging Face Transformers + scikit-learn are your tools here.
   **Structural shortcut:** Many companies use known ATS (Applicant Tracking Systems) platforms — Greenhouse, Lever, Workday, BambooHR, iCIMS, Taleo, SmartRecruiters, Ashby. These have recognizable URL patterns and page structures. Build a fingerprinting system that detects which ATS a site uses, and you can skip the discovery problem entirely for those sites — you already know where the jobs are and what the page structure looks like. This alone covers a surprisingly large percentage of companies.
         
3. Job Listing Identification (The Second Hard Problem)
   Once you're on a careers page, you need to find every individual job listing. The challenge: careers pages have wildly inconsistent structures.
   **DOM analysis approach:** Use BeautifulSoup or lxml to parse the HTML, then look for repeated structural patterns. Job listings almost always appear as repeated sibling elements — a <div> or <li> containing a title, sometimes a location, sometimes a department, with a link to a detail page. Write a "repeating block detector" that finds groups of structurally similar elements. The library extruct can help extract structured data (JSON-LD, Microdata, RDFa) that some sites embed — when present, this is the highest-quality signal.
   **Visual/layout approach:** Render the page and use layout analysis. Tools like readability-lxml strip out nav/footer/sidebar noise. More advanced: use a vision model (LLaVA or a similar open-source multimodal model via Ollama) to look at a screenshot of the page and identify the job listing region. This sounds exotic but it's remarkably effective for handling edge cases.
   **LLM extraction approach:** Feed the cleaned page content (converted to markdown via something like html2text or markdownify) into an LLM with a prompt like: "Identify all job listings on this page. For each, extract the title, URL, and any visible metadata." This is your highest-accuracy approach but also the slowest and most expensive to run. Use it as a verification layer or for pages where your structural approach fails.
   **Hybrid strategy (recommended):** Use structural detection as your primary method, with LLM verification on a sample of results. If structural detection finds 0 jobs on a page you expected to have jobs, escalate to full LLM extraction. Track disagreement rates between methods to identify sites that need manual review.
   **Pagination and infinite scroll:** Many job listings span multiple pages or use lazy loading. Your crawler needs to handle "Load More" buttons (Playwright can click them), infinite scroll (Playwright can scroll and wait for new content), and URL-based pagination (?page=2, ?offset=20). Build a pagination detector that identifies which pattern a site uses.
      
4. Field Extraction & Schema Mapping (The Third Hard Problem)
   This is about going from "here's a job listing" to a structured record with title, company, location, salary, description, requirements, etc.
   **Define your schema first.** Something like:
   * Job title, Company name, Location (city/state/country/remote), Employment type (full-time/part-time/contract), Salary/compensation range, Department/team, Seniority level, Description, Requirements/qualifications, Benefits, Application URL, Date posted, Date closing
   
     ⠀**CSS selector mapping approach:** For each site, identify which CSS selectors or XPath expressions map to which schema fields. This is the "template" approach — once you've mapped a site's structure, re-extraction is fast and reliable. Store these mappings in your database. The challenge is creating them initially and maintaining them when sites change.
     **Auto-mapping with LLMs:** Send the HTML (or cleaned markdown) of a job detail page to an LLM with your schema definition and ask it to extract structured data. Frameworks for this: LangChain has extraction chains, but more directly, use instructor (a Python library that forces LLM output into Pydantic models). Define your job schema as a Pydantic model, and instructor guarantees you get valid structured output.
     **Template learning:** This is the innovative middle ground. Visit a few job pages on the same site, run LLM extraction on each, then compare the DOM structures to learn which HTML elements consistently contain which fields. Effectively, you're using the LLM to bootstrap the CSS selector mappings automatically. Once learned, you switch to fast selector-based extraction for that site. Libraries like lxml.html.diff can help compare page structures.
     **Schema.org and structured data:** Check for JobPosting schema markup (JSON-LD). Google requires this for jobs to appear in Google Jobs, so an increasing number of sites include it. When present, this is perfect structured data handed to you for free. Use extruct to pull it.
     **Validation layer:** Whatever extraction method you use, run validation. Does the job title look like a job title (not a department name or page heading)? Is the location parseable to a real place? Is the salary in a reasonable range? Use rule-based validation plus anomaly detection (a simple outlier model on field lengths, formats, etc.).
   
   5. Scheduling, Change Detection & Re-crawling
   **Scheduling:** Use Celery with Redis as your task queue (both open-source). Define crawl tasks per-site with configurable intervals. Not every site needs the same frequency — high-volume job boards might need daily crawls, while a small company's careers page might only need weekly.
   **Change detection:** Don't re-process pages that haven't changed. Store content hashes (or structural hashes that ignore dynamic elements like timestamps and session tokens) and only trigger extraction when the page has actually changed. This dramatically reduces your processing load.
   **Diffing:** When a page does change, compute what changed. New job listings, removed listings, modified listings. Store historical snapshots to track job lifecycle (posted → active → removed/filled). deepdiff is useful for comparing structured extraction results.
   **Monitoring:** Build a health dashboard that tracks crawl success rates per site, extraction confidence scores, and schema validation pass rates. Sites will break — they redesign, change their ATS, add bot protection. You need to detect this quickly. Airflow or Prefect (both open-source) work well for orchestrating and monitoring these pipelines.

   6. Anti-Bot Considerations
   This is a practical reality you'll face. Many sites use Cloudflare, DataDome, or similar bot protection.
   **Ethical first:** Always respect robots.txt. Set reasonable crawl rates (don't hit a site more than once every few seconds). Use a real User-Agent string. Many sites are perfectly fine with well-behaved crawlers.
   **When sites do block you:** Rotating User-Agent strings helps with basic checks. For more sophisticated protection, curl_cffi (a Python library that mimics real browser TLS fingerprints) gets through most Cloudflare challenges without a full browser. Using Playwright with realistic browser profiles (real viewport sizes, timezone settings, language headers) helps for JS-challenge sites.
   **Residential proxies** are the nuclear option but aren't free. For your open-source stack, consider running Tor as a rotating proxy for sites that block datacenter IPs, though be aware this is slow and some sites block Tor exit nodes.

   7. Future State: Self-Discovery
   You mentioned wanting the system to eventually discover new sites on its own. A few approaches:
   **Outbound link following:** From known job aggregator sites, follow links to company career pages you don't already have in your database. This is a natural crawl expansion.
   **Search engine seeding:** Use a search API (SearXNG is an open-source metasearch engine you can self-host) to run queries like "careers" site:*.com, "we're hiring" [industry], etc. Feed discovered URLs into your pipeline.
   **Company database integration:** Use open datasets (Wikidata, OpenCorporates) to get lists of companies, then attempt to find their websites and careers pages. This gives you systematic coverage of an industry or region.
